连接表达式

连接条件

on：允许在参与连接的关系上设置通用的谓词

select * from student join takes on student.ID = takes.ID；

on表明：如果一个来自 student 的元组和一个来自 takes 的元组在 ID 上的取值相同，那么他们是匹配的

on student.ID = takes.ID    student natural join takes
select * from student join takes where student.ID = takes.ID；

只显示一次ID

select student.ID  as ID , name , dept_name , tot_cred , course_id , sec_id , semester , year , grade from student join takes on student.ID = takes.ID；

on 条件可以表示任何SQL谓词，从而使用on条件的连接表达式就可以表示比自然连接更为丰富的连接条件

使用带on条件的连接表达式的查询可以用不带on条件的等价表达式来替换，只要把on字句中的谓词移到where子句中即可

引入on条件的优点：

  • 外连接：on条件的表现与where条件是不同的
  • on子句中指定连接条件，并在where子句中出现其余的条件，这样的SQL查询更易懂

外连接

显示所有学生的ID、name、dept_name、tot_cred

select * from student natural join takes；
  • 在参与连接的任何一个或两个关系中的某些元组可能会“丢失”

外连接（outer join）运算：在结果中创建空值元组

  • 左外连接

    内连接+左边失配的元组（缺少的右边关系属性用null）

  • 右外连接

    内连接+右边失配的元组（缺少的左边关系属性用null）

  • 全外连接

    内连接 + 左边失配的元组（缺少的右边关系属性用null）+ 右边失配的元组（缺少的左边关系属性用null）

  • 内连接

    舍弃不匹配的元组

左外连接

计算内连接结果

对于在内连接的左侧关系中任意一个与右侧关系中任何元组都不匹配的元组 t，向连接结果中加入一个元组 r，r 的构造如下：

  • 元组 r 从左侧的关系得到的属性被赋为 t 中的值
  • r 的其他属性被赋为 null

查询一门课也没有选修的学生

select ID 
from student natural left outer join takes
where course_id is null;

右外连接

计算内连接结果

对于在内连接的右侧关系中任意一个与左侧关系中任何元组都不匹配的元组 t，向连接结果中加入一个元组 r，r 的构造如下：

  • 元组 r 从右侧的关系得到的属性被赋为 t 中的值
  • r 的其他属性被赋为null

全外连接

左外连接和右外连接的组合

内连接结果计算出来之后，左侧关系中个不匹配右侧关系任何元组的元组被添上空置并加到结果中；右侧关系执行同样操作

查询：显示Comp.Sci.系所有学生以及他们在2009年春季选修的所有课程段的列表

select *
from ( select *
    	from student
    	where dept_name=‘Comp.Sci’)
natural full outer join
(select *
from takes
where semester=‘Spring’ and year=2009);

on 子句可以和外连接一起使用

select * from student left outer join takes on studen.ID = takes.ID ; #ID出现两次

on 和 where 在连链接中的表现是不同的

  • 外连接：为那些对相应内连接结果没有贡献的元组补上null并加入结果
  • on：外连接声明的一部分
  • where子句不是
select * from student left outer join takes where studen.ID = takes.ID ;

视图

让所有用户都看到数据库的整个逻辑模型是不合适的，出于安全考虑，可能需要向用户隐藏特定的数据。

考虑一个职员需要知道教师的标识、姓名和所在院系，但是没有权限看到教师工资值

select ID, name, dept_name
from instructor

虚关系：概念上包含查询的结果，并不预先计算并存储，而是在使用虚关系的时候才通过执行查询被计算出来.

视图：任何像这种不是逻辑模型的一部分，但作为虚关系对用户可见的关系.

在任何给定的实际关系集合上能够支持大量视图

视图定义

SQL中用create view 命令定义视图， 具体的格式为

create view v as < query expression >

是任何合法的 SQL 表达式 。 v 表示视图名称

例如：访问instructor关系中除salary之外的所有数据的职员

create view faculty as
select ID , name , dept_name
from instructor;

Physics系2009年秋季学期所开设的所有课程段列表，其中包括每个课程段在哪栋建筑的哪个房间授课信息

create view physics_fall_2009 as
select course.course_id , sec_id , building , room_number 
from course, section
where course.course_id = section.course_id
        and course.dept_name = ‘Physics’
        and section.semeter = ‘Fall ’
        and section.year = ‘2009’ ;

SQL查询中使用视图

一旦定义了一个视图，就可以用视图名指代该视图生成的虚关系

create view physics_fall_2009 as
select course.course_id, sec_id, building, room_number
from course, section
where course.course_id = section.course_id
	and course.dept_name = ’Physics’
	and section.semester = ’Fall’
	and section.year = ’2009’;

select course_id from physics_fall_2009 where building = ‘Watson’;

视图的属性名可以按下述方式显式指定：

create view departments_total_salary(dept_name, total_salary) as
select dept_name, sum (salary)
from instructor
group by dept_name;

直觉上，在任何给定时刻，视图关系中的元组集是该时刻视图定义中的查询表达式的计算结果。

如果一个视图关系被计算并存储，一旦用于定义该视图的关系被修改，视图就会过期

视图通常这样来实现：

  • 当我们定义一个视图时，数据库系统存储视图的定义本身，而不存储定义该视图的查询表达式的执行结果

一个视图可能被用到定义另一个视图表达式中

create view physics_fall_2009_watson as
	select course_id, room_number
	from physics_fall_2009
	where building= ’Watson’;

create view physics_fall_2009_watson as
(select course_id, room_number
from (select course.course_id, building, room_number
          from course, section
          where course.course_id = section.course_id
               and course.dept_name = ’Physics’
               and section.semester = ’Fall’
               and section.year = ’2009’)
where building= ’Watson’;

物化视图

物化视图（materializing view）: 特定数据库系统允许存储视图关系，但是它们保证：如果用于定义视图的实际关系改变，视图也跟着改变

 create view departments_total_salary(dept_name, total_salary) as
 select dept_name, sum (salary)
 from instructor
 group by dept_name;

保持物化视图一直在最新状态的过程称为物化视图维护（ materializing view maintenance），或者通常简称视图维护（ view maintenance ）

  • 更新后立即视图维护
  • 访问时维护
  • 周期性物化视图更新

视图更新

用视图表达的数据库修改必须被翻译为对数据库逻辑模型中实际关系的修改

例如：视图faculty被提供给一个职员

insert into faculty values( ‘30765’, ‘Green’, ‘Music’);

插入应表示为对instructor关系的插入 salary？

  1. 拒绝插入，返回错误信息
  2. 向 instructor 关系插入元组 ( ‘30765’, ‘Green’, ‘Music’,null)
create view instructor_info as
select ID, name, building
from instructor , department
where instructor.dept_name = department. dept_name;

insert into instructor_info values(‘69987’,’White’,’Taylor’ );

向 instructor 中插入元组 (‘69987’,’White’,null,null )

向 department 中插入元组 (null，’Taylor’，null )

一般不允许对视图关系进行修改

不同的数据库系统指定了不同的条件以允许更新视图关系；参考数据库系统手册

如果定义视图的查询对下列条件都能满足，我们称SQL视图是可更新的。

  • from子句中只有一个数据库关系
  • select子句中只包含关系的属性名，不包含任何表达式、聚集或distinct声明
  • 任何没有出现在select子句中的属性可以取空值；即这些属性上没有not null约束，也不构成主码的一部分
  • 查询中不含有group by或having子句
create view history_instrustors as
select * from instructor where dept_name = ‘History’ ;

事务

事务（transaction）由查询和（或）更新语句的序列组成

SQL标准规定当一条SQL语句被执行，就隐式地开始了一个事务

  • Commit work：提交当前事务
  • Rollback work：回滚当前事务

一个事务或者在完成所有步骤后提交其行为，或者在不能成功完成其所有动作的情况下回滚其所有动作，通过这种方式数据库提供了对事物具有原子性的抽象

默认方式下，每个SQL语句自成一个事务，且一执行完就提交

如果一个事物要执行多条SQL语句，就必须关闭单独SQL语句的自动提交

完整性约束

完整性约束（Integrity constraints）保证授权用户对数据库所做的修改不会破坏数据库的一致性.

一般来说，完整性约束可以是属于数据库的任意谓词

完整性约束通常被看作是数据库模式设计过程的一部分

Not Null约束

not null

宣布名称和预算不为空。

name varchar(20) not null
budget numeric(12,2) not null

not null声明禁止在属性上插入空值

任何可能导致向一个声明为not null的属性插入空值的数据库修改都会产生错误诊断信息

例如：SQL禁止在关系模式的主码中出现空值

unique约束

唯一性 ( A1,A2,…,AmA_1, A_2, …, A_mA1​,A2​,…,Am​)

唯一的规范规定，属性 A1,A2,…,AmA_1, A_2, …, A_mA1​,A2​,…,Am​ 形成一个候选码。

候选码允许为空（与主码相反）

check 子句

check (P)：指定谓词P，关系中的每个元组都必须满足P，其中P是一个谓词

例如：确保学期是秋、冬、春、夏中的一个。

create table section (
    course_id varchar (8),
    sec_id varchar (8),
    semester varchar (6),
    year numeric (4,0),
    building varchar (15),
    room_number varchar (7),
    time slot id varchar (4), 
    primary key (course_id, sec_id, semester, year),
    check (semester in (’Fall’, ’Winter’, ’Spring’, ’Summer’)));

参照完整性

参照完整性（referential integrity）：一个关系中给定属性上的取值也在另一个关系的特定属性集的取值中出现.

例如， 如果 "生物学 "是出现在教师关系中的一个元组中的部门名称，那么在部门关系中就存在一个 "生物学 "的元组。

让 A 是一组属性，让 R 和 S 是两个包含属性 A 的关系，其中 A 是 S 的主码。 让 R 和 S 是两个包含属性 A 的关系，其中 A 是 S 的主码，如果 A 的任何值出现在 R 中，这些值也出现在 S 中，则称 A 是 R 的外码。

create table course (
	course_id   char(5) primary key,
	title             varchar(20),
	dept_name varchar(20) references department)

create table course (
	…
	dept_name varchar(20),
	foreign key (dept_name) references department
	on delete cascade
	on update cascade,)

级联的替代动作：设置null，设置默认值。

SQL中外码参照的是被参照表中的主码属性

references子句

指定的属性列表必须声明为被参照关系的候选码，primary key或unique约束

dept_name varchar(20) references department

当违反参照完整性约束时，通常的处理是拒绝执行导致完整性破坏的操作

在foreign key子句中可以指明：如果被参照关系上的删除或更新动作违反了约束，那么系统必须采取一些步骤通过修改参照关系中的元组来恢复完整性约束，而不是拒绝这样的动作

create table course
    ( …
       foreign key (dept_name) references department
              on delete cascade
              on update cascade,
       …);

事务中对完整性约束的违反

事务可能包括几个步骤，在某一步之后完整性约束也许会暂时被违反，但是后面的某一步也许就会消除这个违反

SQL允许将initially deferred 子句加入到约束声明中；完整性约束不是在事物的中间步骤上检查，而是在事物结束时检查

一个约束可以被指定为可延迟的（deferrable）

默认：立即检查约束

许多数据库实现不支持延迟约束检查

复杂check条件与断言

check子句中的谓词可以是包含子查询的任意谓词

check (time_slot_id in (select time_slot_id from time_slot))

每个课程段都需要有至少一位教师来讲授.

复杂check条件在确保数据完整性时是很有效的，但其检测开销可能会很大

一个断言（assertion）就是一个谓词，表达了我们希望数据库总能满足的一个条件

域约束和参照完整性约束是断言的特殊形式

create assertion <assertion-name> check <predicate>;

创建断言时，要检测其有效性
目前，还没有一个广泛使用的数据库系统支持在check子句的谓词中使用子查询或create assertion结构

SQL的数据类型与模式

SQL中的日期和时间类型

date: 日期，包含（4位数）年、月、日。

Example: date ‘2015-7-27’

time: 一天中的时间，以时、分、秒为单位。

Example: time ‘09:00:30’ time ‘09:00:30.75’

timestamp: 日期加当天时间

Example: timestamp ‘2015-7-27 09:00:30.75’

cast e as t

  • e:字符串（字符串表达式）
  • t：time、date、timestamp

extract（field from d）

  • 从date或time值d中提取单独的域，可以是year、month、day、hour、minute、second

  • 时区信息可以用timezone_hour、timezone_minute来提取

current_date 返回当前日期

current_time 返回当前时间（带时区）

localtime 返回当前的本地时间（不带时区）

current_timestamp（带时区）、localtimestamp（本地日期和时间，不带时区） 返回时间戳

SQL允许在以上所有类型上进行比较运算，也允许在各种数字类型上进行算术运算和比较运算

创建索引

索引（index）：一种数据结构，允许数据库系统高效地找到关系中那些在索引属性上取给定值的元组，而不用扫描关系中的所有元组

create table student
(ID varchar (5),
name varchar (20) not null,
dept_name varchar (20),
tot_cred numeric (3,0) default 0,
primary key (ID))

create index studentID_index on student(ID)；

如果用户提交的SQL查询可以从索引的使用中获益，那么SQL查询处理器救护自动使用索引

给定的SQL查询是选出ID为22201的student元组，SQL查询处理器就会使用studentID_index索引来找到所需元组，而不用读取整个关系

大对象类型

大对象（照片、视频、CAD文件等）以大对象的形式存储。

  • blob（二进制数据的大对象数据类型）：二进制大对象————对象是未解释的二进制数据的大集合（其解释留给数据库系统之外的应用程序）。
  • clob（字符数据的大对象数据类型）：字符大对象————对象是字符数据的大集合。

“定位器”：JDBC应用接口允许获取一个定位器而不是整个大对象；然后用定位器一点一点取出大对象

类型和域的差别

  • 在域上可以声明约束，例如not null，也可以为域类型变量定义默认值，然而在用户定义类型上不能声明约束或默认值。设计用户定义类型不仅是用它来指定属性类型，而且还将它用在不能施加约束的地方对SQL进行过程扩展
  • 域并不是强制类型的。一个域类型的值可以被赋给另一个域类型，只要它们的基本类型是相容的
    check子句应用到域上时，允许模式设计者指定一个谓词，被声明为来自该域的任何变量都必须满足这个谓词
create domain YearlySalary numeric(8,2)
constraint salary_value_test check (value>=29000.00)

使用in子句可以限定一个域只包含指定的一组值：

create domain degree_level varchar(10)
constraint degree_level_test
check (value in (’Bachelors’, ’Masters’,’Doctorate’));

模式、目录与环境

当代数据库系统提供了三层结构的关系命名机制。最顶层由目录（catalog）构成，每个目录都可以包含模式（schema）。关系、视图等SQL对象都包含在模式中

要在数据库上做任何操作，用户（或程序）都必须先连接到数据库。

每个用户都有一个默认的目录和模式

如果用户想访问存在于另外的模式中的关系，而不是该用户的默认模式，必须制定模式的名字
默认目录和模式是为每个链接建立的SQL环境的一部分。环境还包括用户标识

授权

权限的授予与收回

权限所有权限（all privileges）可以用作所有允许权限的简写形式。一个创建了新关系的用户将自动被授予该关系上的所有权限

grant 语句用来授予权限

grant <权限列表>
on <关系名或视图名> to <用户/角色列表>

<用户/角色列表>

  • 用户id
  • public，允许所有有效用户拥有被授予的权限。

授予视图上的权限并不意味着授予基础关系上的任何权限。

权限的授予者必须已经在指定的项目上拥有权限（或者是数据库管理员）。

select: 允许用户修改关系中的任意元组
Example: 在讲师关系上授予用户U1、U2、U3选择授权:

grant select on instructor to U1, U2, U3

insert: 允许用户在关系中插入元组

update: 允许用户修改关系中的任意元组，既可以在关系的所有属性上授予，又可以只在某些属性上授予

delete: 允许用户从关系中删除元组

用户名public：系统的所有当前用户和将来的用户

默认情况下，被授予权限的用户/角色无权把此权限授予其他用户/角色。SQL允许用授予权限来指定权限的接受者可以进一步把权限授予其他用户

SQL授权机制可以对整个关系或一个关系的指定属性授予权限；但不允许对一个关系的指定元组授权

revoke ：回收权限

revoke <权限列表> on <关系名或视图名> from <用户/角色列表>

角色

创建角色：

create role instructor;

角色授予权限：

grant select on takes to instructor;

角色可以授予给用户，也可以授予给其他角色

create role teaching_assistant
grant teaching_assistant to instructor;

角色链

create role dean;
grant instructor to dean;
grant dean to Satoshi;

一个用户或一个角色的权限包括：

  • 所有直接授予用户/角色的权限；
  • 所有授予给用户/角色所拥有角色的权限

当一个用户登录到数据库系统时，在此会话中用户执行的动作拥有所有直接授予该用户的权限以及所有（直接地或通过其他角色间接地）授予该用户所拥有角色的权限。

视图的授权

create view geo_instructor as
(select *
from instructor
where dept_name = ’Geology’);

模式的授权

SQL标准为数据库模式指定了一种基本的授权机制：只有模式的拥有者才能够执行对模式的任何修改

references 权限：允许用户在创建关系时声明外码

grant reference (dept_name) on department to Mariano;

权限的转移

获得了某些形式权限的用户被允许将此权限授权传递给其他用户

默认方式下，被授予权限的用户/角色无权把得到的权限再授予给另外的用户/角色

grant select on department to Amit with grant option;

用户具有权限的充分必要条件是：当且仅当存在从授权图的根（即代表数据库管理员的顶点）到代表该用户顶点的路径

权限的收回

revoke select on department from Amit, Satoshi cascade;
revoke select on department from Amit, Satoshi restrict;
